# E0 Minimal Input

## 1. 目标

`E0` 的目标不是完整训练框架，而是把 overlap-focused preview 的窗口样本稳定转换为模型可直接消费的最小输入。

这一步解决的问题：

1. 不再依赖临时查询脚本和手工选窗
2. 不再把原始 `SampleWindow` 直接塞给模型
3. 先明确“时间偏移 + 数值特征矩阵”的基础协议

## 2. 输入

E0 直接接收：

- `DatasetBuildResult`

也就是说，E0 发生在：

`SortieBundle -> AlignedSortieBundle -> SampleWindow[]` 之后

## 3. 输出

E0 输出的是按窗口组织的最小实验样本：

- `E0ExperimentSample`

每个样本内部包含两条流：

- `physiology`
- `vehicle`

每条流对应：

- 时间偏移序列
- 点所属 measurement 序列
- 统一特征名序列
- 数值特征矩阵
- 被丢弃的非数值字段信息

## 4. 特征规则

### 保留

- 可稳定转成数值的标量字段
- `int / float / bool`
- 可解析为数字的字符串

### 丢弃

- 时间字符串，例如 `date_time`
- 总线原始时钟字符串，例如 `code1001`
- 复合字符串，例如以 `;` 分隔的多值 PPG 片段
- 其他无法稳定数值化的字段

## 5. 特征命名

E0 默认使用：

- `<measurement>.<field>`

例如：

- `eeg.af3`
- `spo2.toi1`
- `BUS6000019110020.code1002`

## 6. 当前边界

E0 当前只保证：

1. 单架次
2. 单窗口或多窗口样本适配
3. 数值特征矩阵输出

E0 当前不保证：

1. 训练集/验证集划分策略定稿
2. 特征归一化定稿
3. 缺失值插补定稿
4. 最终模型输入张量库绑定

## 7. 进入模型阶段的前提

当下面条件成立时，就可以进入模型编写阶段：

1. 至少有一条 overlap-focused preview 路径能稳定生成 `E0ExperimentSample`
2. 每条流的数值特征矩阵维度可重复生成
3. 时间偏移和窗口边界不再依赖手工推断
4. 非数值字段的处理规则已经固定
