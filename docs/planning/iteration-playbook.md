# Iteration Playbook

更新时间：2026-04-21

## 1. 目的

这份文档沉淀跨阶段可复用的推进方法，解决两个常见问题：

1. 单次会话上下文有限，如何在一轮内最大化收敛。
2. 阶段切换时，如何避免“代码完成但证据不完整”或“文档更新但事实不一致”。

## 2. 单轮会话通用节奏

默认按下面顺序执行（除非当前任务明确要求只做其中一段）：

1. 先锁定收口目标：明确本轮是否要达成“阶段收口”还是“阶段内增量”。
2. 先改代码再改文档：先实现与验证，再回写状态，避免文档先行失真。
3. 必须有测试闭环：功能改动对应 UT/runtime 校验，失败就先修再推进。
4. 最后做文档清理：删除被替代文档与重复段落，保持单一事实源。

## 3. 阶段收口通用 Gate

默认只有在以下条件全部满足时，才允许把阶段状态改为“已完成”：

1. 真实可运行证据：至少一组真实数据实跑成功（非仅 mock/离线）。
2. 判定规则可复现：阈值模板或对比判据明确，且报告中可追溯。
3. 代码测试闭环：相关 UT/runtime 全通过。
4. 状态一致性：`AGENTS.md`、`coding-roadmap.md`、阶段收口文档三者结论一致。

若任一条件不满足：

- 阶段状态维持“进行中”，并记录阻塞项与下一步补救路径。

## 4. 文档治理共性规则

采用“单一事实源 + 最小冗余”策略：

1. 阶段状态只在 `docs/planning/coding-roadmap.md` 维护。
2. 阶段收口细节只在 `docs/planning/stage-*-closure-*.md` 维护。
3. `docs/reports` 每阶段只保留 1 份主报告。
4. 旧报告或旧 summary 一旦被新文档完整覆盖，优先删除而非并行维护。

## 5. 测试治理共性规则

采用“域级 suite 合并”策略：

1. 优先按业务域组织测试，而不是按零散文件堆叠。
2. 建议保持 `test_*.py` 总数在 `8-12` 个区间。
3. 新增逻辑优先复用已有 suite；仅在出现新子域时新增文件。
4. 每轮结束至少运行一次全量 discover，确保合并后无隐性回归。

- `CHRONARIS_ENABLE_NUMPY_RUNTIME_TESTS=1 CHRONARIS_ENABLE_TORCH_RUNTIME_TESTS=1 /home/wangminan/env/anaconda3/envs/chronaris/bin/python -m unittest discover -s tests -p 'test_*.py'`
